Relaying Mail through Sympatico Using Postfix <div class="entry"> <p>Customers of <a href="http://www.sympatico.ca/">Sympatico</a>’s high-speed Internet service receive instructions on how to set up their email; however, these settings are aimed at people using Outlook, Netscape, or Eudora. What I needed were instructions on how to configure my local SMTP server, <a href="http://www.postfix.org/">Postfix</a>, with Sympatico. Of course, that information is not available. This document details how I figured out the settings required to make <tt>smtphm.sympatico.ca</tt> happy.</p> <h2>Installation</h2> <p>You need a version of Postfix built with TLS and SASL support enabled. If your distribution doesn’t come with it pre-built follow the instructions at:</p> <ul> <li><a href="http://www.postfix.org/TLS_README.html#build_tls">Building Postfix with TLS support</a></li> <li><a href="http://http//www.postfix.org/SASL_README.html#build_sasl">Building Postfix with SASL authentication support</a></li> </ul> <p>On my Gentoo system I simply did:</p> <pre>emerge -av postfix sasl<br /></pre> <h2>Configuration</h2> <p>I’m assuming that your Postfix configuration directory is <tt>/etc/postfix</tt>. Adjust appropriately to accommodate your system.</p> <p>To use SASL we need a password file containing our user name and password for the server we are connecting to. Per Sympatico’s instructions the server is <tt>smtphm.sympatico.ca</tt>.</p> <pre>cd /etc/postfix<br />mkdir sasl &amp;&amp; cd sasl<br />echo "[smtphm.sympatico.ca] USERNAME@symaptico.ca:PASSWORD" &gt; passwd<br />postmap hash:passwd</pre> <p>The above creates the password file and the hash-based database file that Postfix uses. Of course replace <tt>USERNAME</tt> with your Symaptico user name, and <tt>PASSWORD</tt> with your password (the email password, not the b1 password to access the Internet.)</p> <p>Make sure in your <tt>main.cf</tt> you have configured your relayhost as: <tt>relayhost = [smtphm.sympatico.ca]</tt></p> <p>Finally, add the following lines to your <tt>main.cf</tt> file:</p> <pre># Enable TLS/SASL for the smtphm.sympatico.ca server<br />smtp_use_tls = yes<br />smtp_tls_note_starttls_offer = yes<br />tls_random_source = dev:/dev/urandom<br /><br />smtp_sasl_auth_enable = yes<br />smtp_sasl_password_maps = hash:/etc/postfix/sasl/passwd<br />smtp_sasl_security_options =</pre> <p>Restart Postfix and send a test message. If all went well you should see the successful sending of the message in the Postfix log file.</p> <p><em>Good luck!</em></p> </div>
Configuring your incoming email client for Sympatico/Hotmail: fetchmail <ol> <li> <p>Make sure that your Sympatico email account is a Hotmail one (using the POP3-over-SSL <tt>pophm.sympatico.ca</tt> server), not an older InterMail one (using a regular POP3 server such as <tt>pop1.sympatico.ca</tt>).</p> </li> <li> <p>Find out where X.509/“privacy-enhanced-mail (PEM)” certificates live on your system, e.g., in the <tt>/usr/share/ssl/certs/</tt> directory. The “<tt>locate .pem</tt>” command may help you figure this out, or you can just create a new world-readable directory exclusively for this.</p> </li> <li> <p>The Sympatico/Hotmail POP3-over-SSL (pop3s) server presents a <strong>server</strong> certificate at every secure-socket-layer (SSL) connection establishment; we will want to be able to verify its authenticity. That <strong>server</strong> certificate is itself signed by a certificate authority (CA), Entrust.net, using a self-signed <strong>root</strong> certificate having the “Entrust.net Secure Server Certification Authority” common name (CN). You need to download, install in its own file, and index this <strong>root</strong> certificate.</p> <pre>sh# <strong>cd /usr/share/ssl/certs</strong><br />sh# <strong>wget -O entrust_ssl_ca.pem https://www.entrust.net/downloads/binary/entrust_ssl_ca.cer</strong><br />sh# <strong>chmod 644 entrust_ssl_ca.pem</strong><br />sh# <strong>openssl x509 -in entrust_ssl_ca.pem -fingerprint -subject -issuer -serial -hash -noout</strong><br />MD5 Fingerprint=DF:F2:80:73:CC:F1:E6:61:73:FC:F5:42:E9:C5:7C:EE<br />subject= /C=US/O=Entrust.net/OU=www.entrust.net/CPS incorp. by ref. (limits liab.)/OU=(c) 1999 Entrust.net Limited/CN=Entrust.net Secure Server Certification Authority<br />issuer= /C=US/O=Entrust.net/OU=www.entrust.net/CPS incorp. by ref. (limits liab.)/OU=(c) 1999 Entrust.net Limited/CN=Entrust.net Secure Server Certification Authority<br />serial=374AD243<br />ed524cf5<br />sh# <strong>c_rehash .</strong><br />sh# <strong>file ed524cf5.*</strong><br />ed524cf5.0: symbolic link to `entrust_ssl_ca.pem'<br /></pre> <p>The <tt>c_rehash</tt> command is a Perl script that comes with the OpenSSL toolkit; it creates symbolic links to certificate files (with a “<tt>.pem</tt>” extension) using symlink names based on a hash of each certificate's whole subject name. Fetchmail uses the OpenSSL libraries and will only try to open the installed file by looking it up under the <tt>ed524cf5.0</tt> (or <tt>ed524cf5.1</tt> and so on) file name in the end.</p> <p>The <strong>server</strong> certificate itself does not need to be stored anywhere.</p> </li> <li> <p>The fetchmail client does not need to present a <strong>client</strong> certificate. Instead, client authentication is performed using the <tt>USER</tt> and <tt>PASS</tt> post-office-protocol version 3 (POP3) commands within the SSL connection, once it has been properly established.</p> </li> <li> <p>Edit the appropriate <tt>.fetchmailrc</tt> file, depending on how you run fetchmail (i.e., as <tt>root</tt> or as a lesser-privileged local user). The contents of this file are documented by the fetchmail(1) man page, which can be consulted using the “<tt>man fetchmail</tt>” command. You will need an entry such as this one:</p> <pre>poll pophm.sympatico.ca with proto POP3<br /> user '<em>michael.j.sabia</em>@sympatico.ca' there with password 'foo' is <em>michael</em> here<br /> options ssl sslfingerprint 'B4:C0:69:97:78:DE:A6:59:FD:87:A2:7C:DC:1C:29:ED'<br /> sslcertck sslcertpath /usr/share/ssl/certs<br /></pre> <p>Add any other option according to your preferences. If other family members also have a Sympatico/Hotmail account and this is <tt>root</tt>'s <tt>.fetchmailrc</tt> file, you can add similar three-line <tt>user</tt> subsections under this same <tt>poll</tt> section; the last two lines will redundantly be identical for every Sympatico/Hotmail account.</p> <p>The specified password is that of the Sympatico/Hotmail account, not the local user's password. Since this configuration file contain passwords, it must be non-world-readable:</p> <pre>sh# <strong>chmod 600 $HOME/.fetchmailrc</strong><br /></pre> <p>The MD5 fingerprint is that of the “pophm.sympatico.ca” <strong>server</strong> certificate delivered by the Sympatico/Hotmail POP3-over-SSL server, not that of the “Entrust.net Secure Server Certification Authority” <strong>root</strong> certificate.</p> <p>Note that this will not be enough to perform certificate-revocation-list (CRL) verifications, which fetchmail/OpenSSL does not support; however, this shortcoming is <em>somewhat</em> mitigated by the MD5 fingerprint verification, at the price of a loss in unattended flexibility (should Sympatico/Hotmail wish to change its <strong>server</strong> certificate for some unlikely reason).</p> <p>There is no need to inform fetchmail that the POP3-over-SSL (pop3s) TCP port is 995; it's smart enough to figure it out on its own.</p> <p>Note that POP3-over-SSL is not the same as regular (non-SSL) POP3 with use of the <tt>STLS</tt> command and capability (i.e., RFC 2595's POP3 STARTTLS extension), which Sympatico/Hotmail does not support.</p> </li> <li> <p>You should be able to test this with</p> <pre>sh# <strong>fetchmail -d0 -v pophm.sympatico.ca</strong><br /></pre> <p>once you complete the email-forwarding/email-delivery part of fetchmail's configuration and integration (which is beyond the email-retrieval scope of this help document).</p> </li> <li> <p>The Sympatico/Hotmail POP3-over-SSL server has no requirement regarding the frequency of connections, but it might be reasonable to not do it more than once every five minutes (i.e., 300 seconds). This can be verified by executing the following command:</p> <pre>sh$ <strong>openssl s_client -connect pophm.sympatico.ca:995 -CApath /usr/share/ssl/certs -quiet</strong><br />depth=1 /C=US/O=Entrust.net/OU=www.entrust.net/CPS incorp. by ref. (limits liab.)/OU=(c) 1999 Entrust.net Limited/CN=Entrust.net Secure Server Certification Authority<br />verify return:1<br />depth=0 /C=CA/ST=Ontario/L=Toronto/O=Bell Canada/OU=Bell Sympatico/CN=pophm.sympatico.ca<br />verify return:1<br />+OK BAYC1-PAPOP02 POP3 Server ready<br /><strong>CAPA</strong><br />-ERR invalid command<br /><strong>QUIT</strong><br />+OK BAYC1-PAPOP02 POP3 Server signing off<br />read:errno=0<br /></pre> <p>Make sure to respect this if automatic polling is configured. This can be achieved by specifying “<tt>set daemon 300</tt>” in the <tt>.fetchmailrc</tt> file, or by specifying “<tt>-d 300</tt>” on the fetchmail command line, where 300 can be replaced by a greater number if desired.</p> </li> <li> <p>If your Sympatico/Hotmail “junk e-mail filter” is set to anything other than <strong>low</strong>, much of your legitimate incoming email might mistakenly wind up in your “junk E-Mail” folder instead of your “Inbox” folder. You may have to use the web interface to your Sympatico/Hotmail email account to change this setting.</p> </li> </ol>

SSH login without password <h2>Your aim</h2> <p>You want to use Linux and OpenSSH to automize your tasks. Therefore you need an <strong>automatic</strong> login from host A / user a to Host B / user b. You don't want to enter any passwords, because you want to call <tt>ssh</tt> from a within a shell script.</p> <h2>How to do it</h2> <p>First log in on A as user a and generate a pair of authentication keys. Do not enter a passphrase:</p> <pre class="shell">a@A:~&gt; ssh-keygen -t rsa<br />Generating public/private rsa key pair.<br />Enter file in which to save the key (/home/a/.ssh/id_rsa): <br />Created directory '/home/a/.ssh'.<br />Enter passphrase (empty for no passphrase): <br />Enter same passphrase again: <br />Your identification has been saved in /home/a/.ssh/id_rsa.<br />Your public key has been saved in /home/a/.ssh/id_rsa.pub.<br />The key fingerprint is:<br />3e:4f:05:79:3a:9f:96:7c:3b:ad:e9:58:37:bc:37:e4 a@A<br /></pre> <p>Now use <tt>ssh</tt> to create a directory <tt>~/.ssh</tt> as user b on B. (The directory may already exist, which is fine):</p> <pre class="shell">a@A:~&gt; ssh b@B mkdir -p .ssh<br />b@B's password: <br /></pre> <p>Finally append a's new public key to <tt>b@B:.ssh/authorized_keys</tt> and enter b's password one last time:</p> <pre class="shell">a@A:~&gt; cat .ssh/id_rsa.pub | ssh b@B 'cat &gt;&gt; .ssh/authorized_keys'<br />b@B's password: <br /></pre> <p>From now on you can log into B as b from A as a without password:</p> <pre class="shell">a@A:~&gt; ssh b@B hostname<br />B<br /></pre> <p><strong>A note</strong> from one of our readers: Depending on your version of SSH you might also have to do the following changes:</p> <ul> <li>Put the public key in <tt>.ssh/authorized_keys2</tt></li> <li>Change the permissions of <tt>.ssh</tt> to <tt>700</tt></li> <li>Change the permissions of <tt>.ssh/authorized_keys2</tt> to <tt>640</tt></li> </ul>
